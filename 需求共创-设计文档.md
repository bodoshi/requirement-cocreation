# 知识库体系：使用模型与整合方案

## Context

我们已经产出了四个知识库的初始内容（①需求框架库文档、②细节规范库文档、③业务规则库Bitable、映射分析文档），但用户反馈**不知道这些该怎么用**。

根本原因：我们做了"从优质PRD提取知识"这一步，但跳过了"从check-prd迁移知识"和"定义使用路径"两步。导致新文档和旧check-prd的关系不清，AI消费路径不明，人类查阅入口缺失。

---

## 一、问题诊断：为什么不知道怎么用

### 现状

| 资产 | 内容来源 | 当前状态 |
|------|---------|---------|
| check-prd (~1900行) | 审查理论 + 经验总结 | 可用，但是单体 prompt |
| ① 需求框架库文档 | 3份优质PRD提取 | 有内容，无接入点 |
| ② 细节规范库文档 | 3份优质PRD提取 | 有内容，无接入点 |
| ③ 业务规则库 Bitable | 监管文档拆解 | 36条记录，可查询 |
| ④ 系统实现库 | 未建 | — |

### 核心矛盾

```
check-prd 里有 11 维度 checklist + 交互规范 + 字段规范
          ↕ 关系不明
新 ①② 文档里有从真实PRD提取的结构范本 + 设计模式
```

新文档是 check-prd 的**补充**而非**替代**，但这个关系没有定义清楚。

### 四个困惑的答案

1. **和 check-prd 的关系**：check-prd 最终会被"解体"——编排逻辑变成 Skill prompt，知识内容分流到 ①②③④。新文档是从"实战样本"方向往知识库填内容，check-prd 是从"审查理论"方向填内容，两者汇合。
2. **内容粒度**：对 AI 消费来说粒度合适（AI 能处理长文档）；对人类查阅来说需要一层**索引**。
3. **AI 怎么消费**：/PRD内审 运行时读飞书文档 → 提取相关规范 → 与 PRD 交叉比对。但 Skill 还没开发，所以目前 AI 还没在消费。
4. **写 PRD 时怎么查阅**：目前缺少一个"入口"告诉你"写列表页时看②的第一节，写分期方案时看①的第四节"。

## 二、知识全景图：什么内容在哪里、谁消费

### 2.1 知识的两个来源

```
来源A：check-prd（审查理论）          来源B：优质PRD样本（实战经验）
  ├── 11维度 checklist                  ├── PRD三板斧结构
  ├── 交互/字段/设计规范                 ├── 筛选分层/五态模型/OGS...
  ├── 合规检查提示                      ├── 6W1H跨市场分析框架
  └── 评分规则/报告模板                  └── CDD/EDD/不动账户规则
         │                                    │
         └──────── 汇合到同一套知识库 ─────────┘
```

### 2.2 四个知识库的完整定位

| 知识库 | 回答什么问题 | 谁写入 | 谁消费 |
|--------|------------|--------|--------|
| ① 需求框架库 | **PRD 应该长什么样** — 结构、维度、checklist | check-prd迁移 + PRD样本提取 | AI审查 + PM写PRD前查阅 |
| ② 细节规范库 | **某个组件该怎么设计** — 交互模式、字段规格 | check-prd迁移 + PRD样本提取 | AI审查 + PM写到具体功能时查阅 |
| ③ 业务规则库 | **这个市场有什么合规要求** — 监管规则、参数 | 监管文档拆解 | AI审查 + PM涉及合规时查阅 |
| ④ 系统实现库 | **现在系统是怎么实现的** — 架构、约束、可复用组件 | 技术文档整理 | AI审查 + PM做技术方案时查阅 |

### 2.3 当前每个知识库里有什么

| 知识库 | 已有内容（来源B） | 缺失内容（来源A，待从check-prd迁移） |
|--------|------------------|-----------------------------------|
| ① 需求框架库 | 三板斧/合规争议/6W1H/MVP分期/ER描述（6个框架） | **11维度checklist**（~70+检查项）、PRD类型侧重映射、常见问题诊断表 |
| ② 细节规范库 | 筛选分层/批量三态/版本五态/多语言/审核工作台/热更新/主副栏目/存档（8个规范） | **弹出层选择/表单模式/操作反馈/消息系统/字段规格**（~20+规范） |
| ③ 业务规则库 | 36条Bitable记录（开户+不动账户+尽调） | 更多业务模块的规则（出入金、品类激活等） |
| ④ 系统实现库 | 无 | 全部待建 |

**关键发现**：①② 目前只有来源B的内容（~30%），缺来源A的内容（~70%）。所以它们看起来像"零散的补充材料"而非"完整的参考手册"。

## 三、两条使用路径

### 3.1 路径一：AI 消费（/PRD内审 Skill）

```
用户输入 PRD 链接
    │
    ▼
Skill prompt（编排层）──── 固化在 Skill 内：
    │                     · 产品定型逻辑
    │                     · 执行阶段顺序
    │                     · 评分规则 + 报告模板
    │                     · R1-R7 风险项定义
    ▼
按维度读取知识库 ────────── 运行时动态读取：
    │  ① 读 checklist → "PRD是否有XX"
    │  ② 读规范 → "列表页设计是否符合筛选分层规范"
    │  ③ 读规则 → "方案是否满足SFC合规要求"
    │  ④ 读实现 → "是否与现有架构冲突"
    ▼
输出审查报告
```

**当前状态**：Skill 未开发，AI 消费路径不可用。
**前置条件**：①② 内容需要补完（迁移 check-prd），才值得开发 Skill。

### 3.2 路径二：人类查阅（PM 写 PRD 时）

**需要一个"入口索引"**——告诉 PM 在什么场景下看什么内容：

| 我正在做的事 | 看哪个库 | 看哪一节 |
|-------------|---------|---------|
| 开始写一份新 PRD | ① 需求框架库 | 三板斧（确定用哪些结构板块） |
| 写效果预期 / 项目目标 | ① 需求框架库 | 板斧二：效果预期模板 |
| 做多市场方案对比 | ① 需求框架库 | 6W1H 跨市场分析框架 |
| 定分期计划 | ① 需求框架库 | MVP 分期决策矩阵 |
| 设计一个列表页 | ② 细节规范库 | 筛选分层 + 默认排序 |
| 设计批量操作 | ② 细节规范库 | 批量操作三态反馈 |
| 设计版本管理功能 | ② 细节规范库 | 版本管理五态模型 |
| 设计多语言内容 | ② 细节规范库 | 多语言设计规范 |
| 设计审核流程 | ② 细节规范库 | 审核工作台 OGS + 操作规范 |
| 标注可配置参数 | ② 细节规范库 | 热更新字段区分 |
| 涉及合规 / 监管 | ③ 业务规则库 | 按业务模块 + 市场查 Bitable |
| 有合规争议 | ① 需求框架库 | 合规争议处理框架 |

**当前状态**：入口索引不存在。PM 要自己翻文档找，体验差。

## 四、推荐方案：先让知识库可用，再开发 Skill

### 方案对比

| 路径 | 做什么 | 投入 | 效果 |
|------|--------|------|------|
| A. 先补完知识库 | 把 check-prd 内容迁移到 ①② + 加入口索引 | 中 | ①② 变成完整参考手册，人类立刻可用 |
| B. 先开发 Skill | 基于当前不完整的 ①② 开发 /PRD内审 | 大 | AI 能跑但检查不全面，不如直接用 check-prd |
| C. 不迁移，直接在 Skill 里引用 check-prd + 知识库 | Skill 同时读 check-prd 原文 + 四个知识库 | 小 | 能快速跑通，但 check-prd 仍是单体黑盒 |

**推荐路径 C → A 渐进式**：

1. **立刻做（本次）**：给 ①② 加入口索引，让人类现在就能用
2. **短期做**：开发 /PRD内审 Skill，初版同时读 check-prd 原文 + 知识库做交叉验证
3. **中期做**：逐步把 check-prd 内容拆到 ①②，每拆一块，Skill 少读一块 check-prd
4. **最终态**：check-prd 退役，Skill 完全从知识库读取

### 本次实施内容

**产出 1：给 ①② 各加一页"使用指南"**

在每份飞书文档的开头加一节"使用指南"，包含：
- 这份文档是什么、不是什么
- 使用场景速查表（"我正在做X → 看第Y节"）
- 和 check-prd 的关系说明

**产出 2：更新 check-prd prompt，增加知识库引用指令**

在 check-prd 的执行指南中加一段：
- 检查维度 08 交互设计时，额外参考 ② 细节规范库
- 检查维度 01 合规要求时，额外参考 ③ 业务规则库 Bitable
- 这样 check-prd 立刻就能"消费"知识库内容

**产出 3（可选）：创建一份"知识库全景索引"飞书文档**

一页纸说清楚四个知识库分别是什么、在哪里、怎么找到需要的内容。

### 验证方式

1. 让用户打开 ② 细节规范库文档，看使用指南是否能在 30 秒内找到"设计列表页该看哪里"
2. 用 check-prd + 知识库引用指令跑一份真实 PRD，看是否比纯 check-prd 多发现问题

## 五、`/需求共创` 交互式对话 Skill 设计

### 5.0 Context

用户提供了团队实际使用的 PRD 模板（`HB8sdQXYToJIWqxT7XrcMtGbnGe`），指出 PM 经常跳过 §3.4 合规checklist、§4.1 合规争议、§4.3 数据存档、§4.5 发布计划等模块。用户希望 AI 以**交互式对话**方式主动引导 PM 确认这些模块，并**提供具体建议**（而非让 PM 被动自检）。

### 5.1 定位区分

| Skill | 触发时机 | 核心动作 | 输出 |
|-------|---------|---------|------|
| `/需求共创` | PM 写完 PRD 初稿后 | 交互式对话，主动补全模板中被跳过的模块 | PM 可直接粘贴到 PRD 的内容片段 |
| `/PRD内审` | PRD 准备评审前 | 11维度系统性审查 | 评分报告 + 问题清单 |

`/需求共创` 是"帮你写完"，`/PRD内审` 是"帮你检查"。两者串联使用：先共创补全 → 再内审打分。

### 5.2 执行流程

```
PM 输入：PRD 飞书链接
         │
    ┌─────┴─────┐
    │ Phase 1   │  扫描 + 识别
    │ Intake    │  
    └─────┬─────┘
          │  · 读取 PRD 全文
          │  · 识别业务域（开户/资金/合规/账户管理）
          │  · 识别适用市场（HK/JP/SG/MY/AU/CA）
          │  · 检测模板各节的填写状态（空白/简略/充分）
          ▼
    ┌─────┴─────┐
    │ Phase 2   │  知识匹配
    │ Context   │
    └─────┬─────┘
          │  · 按业务域 → 激活对应 domain checklist（3.2/3.3/3.4/3.5）
          │  · 按适用市场 → 查 ③ Bitable 获取相关监管规则
          │  · 按 PRD 涉及的设计组件 → 匹配 ② 中的规范条目
          │  · 按 PRD 范围 → 匹配 ① 中的框架（争议/分期/存档）
          ▼
    ┌─────┴─────┐
    │ Phase 3   │  交互式对话（核心）
    │ Dialogue  │
    └─────┬─────┘
          │  对每个"空白/简略"模块：
          │  ① 解释为什么这个模块和你的 PRD 相关
          │  ② 基于知识库给出具体建议
          │  ③ 请 PM 确认 / 调整 / 标记"不适用"并说明理由
          │  ④ 根据 PM 回答生成可粘贴的内容片段
          ▼
    ┌─────┴─────┐
    │ Phase 4   │  输出
    │ Output    │
    └─────┬─────┘
          │  · 每个模块的填写建议（可直接粘贴到 PRD）
          │  · 标记"不适用"的模块及理由
          │  · 后续建议（"建议跑 /PRD内审 做完整审查"）
```

### 5.3 四个高频遗漏模块的 AI 对话设计

#### §3.4 合规 Checklist（10 项自检）

**当前问题**：PM 一路勾"不涉及"，没有真正思考。

**AI 对话模式**：
```
AI: 你的 PRD 涉及 [开户流程变更]，适用市场是 [HK]。
    我从业务规则库中找到以下与你直接相关的监管要求：

    📌 SFC 要求：开户时必须验证身份证件有效期（规则 R-HK-002）
    📌 PDPO 要求：收集个人资料须有明确用途说明（规则 R-HK-010）

    基于这些规则，以下 3 项 checklist 需要你特别确认：

    1️⃣ "是否涉及数据和个人信息相关问题" → 你的方案改变了用户资料提交方式，
       这很可能涉及"用户数据收集"和"跨境传输"。
       建议：确认新流程是否改变了数据存储位置或传输路径。
       你的处理：___

    2️⃣ "是否涉及对协议/风披的变动" → 你新增了 XX 字段，
       如果这个字段出现在客户协议中，可能需要通知存量客户。
       建议：和法务确认是否触发协议变更通知义务。
       你的处理：___

    3️⃣ "是否涉及需要报备监管的功能变动" → 根据 ③，
       HK 开户流程此前已向 SFC 报备，本次变更可能需要补报。
       建议：和合规确认是否需要 SFC 补报。
       你的处理：___

    其余 7 项经分析与你的 PRD 关联度低，默认标记"不涉及"，
    如有疑虑可逐项确认。
```

**知识源**：③ 业务规则库（按市场+业务模块查 Bitable）+ 模板自带 checklist 题目

### 5.4 通用 Checklist（§3.1）的增强

除了四个高频遗漏模块，§3.1 通用 checklist 虽然 PM 会看，但填写质量参差。AI 可以：

- 读取 PRD 中的需求细节 → 自动判断哪些通用 checklist 项与当前 PRD 相关
- 对相关项给出具体化建议（不只是"已考虑"，而是"考虑了什么"）

### 5.7 设计决策（已确认）

**决策 1：建议权威性 → 分层定位 + 全程标注"AI建议"**
- 所有建议都标注为"AI建议"
- ③ 中有明确法规原文的 → 直接引用，标"规则引用"
- 判断模糊的（如"是否构成争议"）→ 标"建议与合规确认"
- 执行原则：清晰度梯度标注，不让 PM 误以为 AI 判断等于合规结论

**决策 2：对话粒度 → 智能混合**
- AI 判断"与本 PRD 高度相关"的项 → 逐项引导（附背景+建议+请确认）
- AI 判断"关联度低"的项 → 批量标记"不涉及"，PM 只需扫一眼
- 实现方式：Phase 2 知识匹配后给每个 checklist 项打相关度分

**决策 3：覆盖范围 → 业务域路由 + 需求细节交叉检查**

覆盖三个层次：

```
层次一：Checklist 路由（按业务域激活）
┌──────────────┬─────────────────────────┐
│ 业务域        │ 激活的 Checklist         │
├──────────────┼─────────────────────────┤
│ 开户          │ §3.1 通用 + §3.2 开户    │
│ 资金          │ §3.1 通用 + §3.3 资金    │
│ 合规          │ §3.1 通用 + §3.4 合规    │
│ 账户管理      │ §3.1 通用 + §3.5 账户管理 │
│ 跨域          │ §3.1 通用 + 多个域 checklist │
└──────────────┴─────────────────────────┘

层次二：必备确认事项（AI 判断辅助确认）
  §4.1 合规争议 → AI 基于 PRD 内容 + ③ 主动识别潜在争议
  §4.3 数据存档 → AI 基于 PRD 实体 + ② 推荐存档策略
  §4.5 发布计划 → AI 基于 PRD 范围 + ① 推荐灰度策略
  §4.2/4.4/4.6 → AI 判断是否涉及，涉及则提醒

层次三：需求细节交叉检查（核心增值）
  读取 §2 需求细节 → 与知识库交叉比对：
  · 涉及列表页设计 → 引用 ② 筛选分层规范
  · 涉及审核流程 → 引用 ② 审核工作台 OGS 规范
  · 涉及版本管理 → 引用 ② 版本管理五态模型
  · 涉及多市场 → 引用 ① 6W1H 跨市场分析框架
  · 涉及分期规划 → 引用 ① MVP 分期范本
  · 涉及数据实体 → 引用 ② 数据存档规范
  · 涉及合规要求 → 查 ③ Bitable 获取具体规则

层次四：跨模块关联检查（§5.14，新增）
  提取 PRD 中的实体/接口/状态/依赖 → 五种模式扫描：
  · D1 共享数据实体变更 → 列出所有消费方确认影响
  · D2 上下游系统联动 → 确认接口/写回/触发设计
  · D3 跨系统状态一致性 → 定义优先级和展示策略
  · D4 供应商/外部依赖共享 → 确认数据是否打通
  · D5 功能边界模糊 → 明确系统间职责分工
```

### 5.9 需求细节交叉检查的匹配规则

| PRD 中出现的设计元素 | 识别关键词 | 匹配的知识库条目 | AI 建议示例 |
|---------------------|----------|----------------|-----------|
| 列表页/查询页 | 列表、筛选、搜索、查询 | ② §1 筛选分层 + 默认排序 | "你的列表有6个筛选项，建议3个常驻+3个高级筛选" |

**OKR 数据（当前已知）：**

| 业务模块 | Objective | Key Results（⭐标记） |
|---------|-----------|---------------------|

| 合规业务 | 提高合规系统营运人效 | ⭐降低供应商采买成本 |
| 合规业务 | 提高合规系统集团可复用性 | （待补充 KR） |
| 合规业务 | 降低合规场景系统风险 | （待补充 KR） |
| 登录注册 | 增强登录场景的安全验证因子 | （待补充 KR） |
| 登录注册 | 提高注册转化率 | （待补充 KR） |

**存储方式**：建议用 Bitable 新表（或在现有 ③ Bitable 中新增一个"OKR"表），结构为：

| 字段 | 类型 | 说明 |
|------|------|------|
| 业务模块 | 单选 | 账户管理/合规业务/登录注册/开户/资金/... |
| 层级 | 单选 | O / KR |
| 内容 | 文本 | OKR 描述 |
| 上级目标 | 双向关联 | KR→O 关联 |
| 周期 | 单选 | 2026H1 / 2026H2 / ... |
| 已关联PRD数 | 数字 | 回溯统计用 |

**AI 对话模式**：
```
AI: 你的 PRD 涉及 [合规业务] 模块。当前该模块的 OKR 如下：

    📌 O1: 提高合规系统营运人效
       └ KR1: 降低供应商采买成本
    📌 O2: 提高合规系统集团可复用性
    📌 O3: 降低合规场景系统风险

    你的需求主要挂钩哪个 O？对应哪个 KR？
    → 这将作为 PRD 的效果聚焦指标。
    你的选择：___
```

**双向价值**：
- **正向**：写 PRD 时明确"我这个需求对 OKR 的贡献是什么"，避免做与 OKR 无关的事
- **逆向**：按 OKR 维度汇总所有关联的 PRD，看"这个 O 到底投入了多少需求、效果如何"

**输出落地**：OKR 对齐信息必须出现在最终生成的需求单/PRD 中（不只是验证过程中确认），格式示例：

```
┌ OKR 挂靠 ──────────────────────────────┐
│ 业务模块：合规业务                        │
│ 目标(O)：提高合规系统营运人效              │
│ 关键结果(KR)：降低供应商采买成本           │
│ 本需求预期贡献：XX审核耗时从Y降至Z         │
└────────────────────────────────────────┘
```

#### 5.10.3 三问法验证（核心）

读取 PRD §1 背景，逐一验证三个问题：

| 问题 | 验证什么 | 合格标准 | 缺陷信号 |
|------|---------|---------|---------|
| Q1 根目的 | 为什么做这件事？和哪个业务指标挂钩？ | 能指向具体指标（如转化率/合规达标率/处理时效） | "提升用户体验"等模糊表述、无指标关联 |
| Q2 客群+问题 | 给谁用？解决什么问题？有数据/用研支撑吗？ | 有明确用户群体 + 问题描述有数据佐证 | 没有用户分群、问题来源是"老板说的"、无数据支撑 |
| Q3 ROI | 投入多少资源？预期回报是什么？值不值得做？ | 有投入估算 + 回报预期（即使粗略） | 完全没有投入产出分析、或"战略需要"一笔带过 |

**四种变体适配**（来源：新人培训文档）：

| PRD 驱动类型 | 三问法变体 | Q1 侧重 | Q2 侧重 | Q3 侧重 |
|-------------|----------|---------|---------|---------|
| 用户反馈驱动 | 标准三问 | 反馈背后的真实问题 | 反馈用户是否代表目标群体 | 修复成本 vs 影响面 |
| 业务项目驱动 | 项目三问 | 项目的业务目标 | 项目服务的内部/外部客群 | 项目资源投入 vs 业务价值 |
| 老板需求驱动 | 溯源三问 | 老板期望达成的指标 | 老板的判断依据是否验证 | 有无更轻量的替代方案 |
| 数据分析驱动 | 数据三问 | 数据指向的核心问题 | 受影响的用户范围和频次 | 修复后指标提升预期 |

#### 5.10.3 三看六不要（决策健康度检查）

**六不要反面检查** — 逐一扫描 §1 背景，检测是否踩中常见陷阱：

| 编号 | 陷阱 | 检测信号 | AI 对话示例 |
|------|------|---------|-----------|
| N1 | 感觉不错就做 | §1 只有主观判断，无数据 | "你的背景描述了直觉判断，但没有数据支撑。建议补充：[具体数据建议]" |

#### 5.10.4 需求分类 → 后续深度路由

|---------|---------|---------------|

### 5.13 需求文档质量方法论（① 需求框架库 + Phase 3a 增强）

> 来源：`HcxcdgKrHoiTYfxJkuYcseSfn0c`（产品文档质量提升 - sharoncai, 2022）+ `Bn1zdQdqVobqeBxw9RmcLkqCnph`（wiki `OFwHweMOQiBnIZk1gqtcsJisn5e`，如何写好一篇需求文档 - JP市场PM分享）

#### 5.13.1 核心概念：产品需求 vs 产品需求单

| | 产品需求 | 产品需求单 |
|--|---------|----------|
| 回答 | **What is it?**（是什么、为什么、值不值） | **How to make it?**（怎么做、细节逻辑） |
| 比喻 | 装修设计图纸（全屋功能分区+风格+效果） | 施工图纸（水电管道+开关布局+工程分解） |
| 受众 | 上级（价值/成本/收益）、关联方（影响/配合） | 研发（技术可执行性）、测试、业务 |
| 质量权重 | **60%** | **40%** |

**对 `/需求共创` 的意义**：
- Phase 0 主要验证**产品需求质量**（方向对不对、值不值得做）
- Phase 3 主要验证**需求单质量**（写得够不够清楚、能不能开发）
- 两者分开检查，权重不同

#### 5.13.3 需求单七项质量标准

用于 Phase 3a 的**文档质量维度**检查：

| 标准 | 定义 | AI 检测方式 |
|------|------|-----------|
| **无歧义** | 术语唯一定义，无多义词 | 检查是否有名词释义；识别模糊用语（"可能"、"大概"、"待定"） |
| **完整** | 全部功能描述清楚，含合法/非法输入响应 | 检查模板各节填写状态（Phase 1 已做） |
| **明确** | 无"待定"/"TBD"，所有描述确定 | 扫描文档中的待定标记 |
| **结构清晰** | 从文档结构可见产品结构 | 检查标题层级、模块划分是否合理 |
| **逻辑流畅** | 有主题→有重点→段落关联→表达统一 | 检查描述顺序是否递进（前文不引用后文概念） |
| **避免冗余** | 同一功能不重复，语言精练 | 检查是否有重复描述的功能点 |
| **可追踪** | 需求源流清晰可追溯 | 检查是否标注需求来源（用户反馈/数据/业务方） |

**AI 对话模式**：
```
AI: 📋 文档质量检查：

    ✅ 结构清晰：标题层级合理，模块划分清楚
    ✅ 可追踪：需求来源标注了"合规部反馈+用户投诉数据"

    ⚠️ 歧义风险：
    · §2.3 中"适当的"验证方式 — "适当"如何定义？建议明确具体标准
    · §2.5 中出现了"XX系统"但未在名词释义中解释

    ⚠️ 待定项：
    · §2.7 标注了"待确认"，建议在评审前确认或标注决策时间

    ⚠️ 冗余：
    · §2.3 和 §2.6 对同一校验逻辑描述了两次，建议合并或引用
```

### 5.14 跨模块关联检查（Phase 3a+ 新维度）

> **来源**：分析三份优质 PRD 后提取的跨模块依赖模式。用户指出"更新国家代码需要同时更新开户和合规的底层信息"，这类跨模块影响是 PM 常遗漏的盲区。

#### 5.14.1 五种跨模块依赖模式

| 模式 | 定义 | 实战案例（来源 PRD） |
|------|------|---------------------|
| **D1 共享数据实体变更** | 一个系统改了共享实体（国家代码/证件类型/用户资料），其他依赖方必须同步 | 尽调系统：国家风险涉及国籍/居住国/税务国/出生国，改了国家代码需同步开户+合规底层 |
| **D2 上下游系统联动** | 一个系统的输出是另一个系统的输入，变更后下游需适配 | 问卷系统：7+下游消费方（CSS/ECDD/NSS/CDD/开户审批/用户处置/尽调/RPQ），问卷结构变更影响全链路 |
| **D3 跨系统状态一致性** | 用户/账户在多个系统中有状态，需保持一致或定义优先级 | 不动账户系统：审计详情页需联动查看"限制平台状态+CDD状态+自查状态"，状态不一致会误导操作 |
| **D4 供应商/外部依赖共享** | 多个系统依赖同一外部供应商或数据源 | 尽调系统：SG 的 World Check 数据在开户和尽调系统间"很难打通"，同一供应商数据被分割使用 |
| **D5 功能边界模糊** | 功能在系统间的归属不清，导致重复建设或两不管 | 尽调系统：JP反馈"开户和合规分开两个系统，操作麻烦"；开户/尽调都能做风险审核 → 边界需明确 |

#### 5.14.2 AI 检测信号与关键词

| 依赖模式 | PRD 中的检测信号 | 关键词/模式 |
|---------|----------------|-----------|
| D1 共享数据实体 | PRD 中出现"读取XX系统的数据"、"依赖XX字段" | 共享、引用、同步、字段来源、主数据、基础数据 |
| D2 上下游联动 | PRD 中出现"下游系统"、"提供给XX使用"、"触发XX流程" | 下游、上游、消费方、输出到、接口、回调、写回 |
| D3 状态一致性 | PRD 中出现多个状态字段、跨系统查询 | 状态、联动、一致性、同步状态、跨系统查询 |
| D4 供应商共享 | PRD 中出现外部供应商、第三方接口 | 供应商、World Check、KYCC、ASF、第三方 |
| D5 边界模糊 | PRD 中出现"也可以在XX系统做"、功能重叠描述 | 边界、归属、重叠、统一入口、哪个系统负责 |

#### 5.14.3 集成到 Phase 3a 的执行方式

```
Phase 3a 交叉检查中增加"跨模块关联扫描"步骤：

1. 实体扫描：提取 PRD 中涉及的核心数据实体
   → 检查每个实体是否为共享实体（多系统使用）
   → 如果是 → 触发 D1 检查

2. 接口扫描：提取 PRD 中的系统交互（读/写/调用）
   → 检查是否有上下游依赖
   → 如果有 → 触发 D2 检查

3. 状态扫描：提取 PRD 中的状态字段和流转
   → 检查是否有跨系统状态关联
   → 如果有 → 触发 D3 检查

4. 依赖扫描：提取 PRD 中的外部依赖
   → 检查是否有供应商被多系统共享
   → 如果有 → 触发 D4 检查

5. 边界扫描：检查 PRD 功能是否和现有系统有重叠
   → 如果有 → 触发 D5 检查
```

### 5.15 知识沉淀机制（Phase 5：沉淀回流）

> **定位**：六步闭环的最后一环"沉淀回流"。每次 `/需求共创` 运行后，AI 自动识别值得沉淀的新知识，暂存到审核队列，由负责人审核后入库。这是知识库持续生长的核心引擎。

#### 5.15.1 沉淀触发点

`/需求共创` 的每个 Phase 都可能产生沉淀候选：

| Phase | 沉淀类型 | 示例 | 建议归属 |
|-------|---------|------|---------|
| Phase 0 | 新的 OKR 对齐模式 | 发现某类需求普遍挂不上 KR，说明 OKR 体系有盲区 | OKR Bitable（标注"待补充KR"） |
| Phase 3a 交叉检查 | PRD 中的优质设计模式（知识库未收录） | PM 设计了一种新的审批超时递进机制，比现有规范更完善 | ② 细节规范库 |
| Phase 3a 跨模块 | 新发现的系统间依赖关系 | 发现"风控系统"和"开户系统"共享"国家风险等级"实体 | ④ 系统实现库 |
| Phase 3b checklist | PM 的合规决策和理由 | PM 确认"HK 开户简化流程属于指导性要求，当前可用" | ③ 业务规则库（作为产品解读） |
| Phase 3b 必备事项 | 新的关联事项类型（9项之外） | 发现某类需求普遍需要"合规培训"作为上线前置 | Skill prompt 更新（扩展关联事项） |
| Phase 4 输出 | 跨 PRD 的共性问题 | 连续 3 份 PRD 都缺少灰度观测指标 | ① 需求框架库（强化该检查项） |

#### 5.15.2 AI 识别规则（什么值得沉淀）

AI 在 Phase 4 输出完成后，自动扫描本次会话，按以下标准判断"值得沉淀"：

| 标准 | 判断条件 | 沉淀信号 |
|------|---------|---------|
| **通用性** | 这个发现/模式是否适用于其他 PRD？ | 不是个例才沉淀 |
| **新颖性** | 知识库中是否已有类似内容？ | 已有则跳过或标"更新建议" |
| **可操作性** | 能否转化为具体的检查项/规范/规则？ | 能落地才沉淀，不沉淀模糊感悟 |
| **来源可靠** | 是 PM 确认的决策还是 AI 推测？ | PM 确认过的优先 |

**AI 输出格式**（Phase 4 末尾附加）：

```
📋 知识沉淀候选（本次共创发现 X 条值得沉淀的知识）：

| # | 内容摘要 | 建议归属 | 沉淀理由 | 来源Phase |
|---|---------|---------|---------|----------|
| 1 | 审批超时三级递进机制（1h提醒→4h升级→24h自动转交） | ② 细节规范库 | 比现有规范更完善，适用于所有审核场景 | 3a |
| 2 | HK开户简化流程属于SFC指导性要求（非红线） | ③ 业务规则库 | 补充R-HK-003的产品解读层 | 3b |
| 3 | 风控系统与开户系统共享"国家风险等级"实体 | ④ 系统实现库 | 新发现的跨模块依赖（D1模式） | 3a+ |

是否将以上候选提交到沉淀暂存表？（Y/N/逐条确认）
```

#### 5.15.3 沉淀暂存表设计（Bitable）

在现有 ③ 业务规则库 Bitable（`LORwb36tYaVqLLsZ3czcEP0OnUh`）中**新增一个数据表**"知识沉淀暂存"：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 内容摘要 | 文本（索引） | 沉淀知识的简要描述 |
| 详细内容 | 文本 | 完整的知识描述，含具体参数/规则/模式 |
| 来源PRD | 超链接 | 产出该知识的 PRD 飞书链接 |
| 来源Phase | 单选 | Phase 0 / 3a / 3a+跨模块 / 3b / 4 |
| 建议归属 | 单选 | ①需求框架库 / ②细节规范库 / ③业务规则库 / ④系统实现库 / Skill更新 |
| 沉淀理由 | 文本 | 为什么值得沉淀（通用性+新颖性说明） |
| 状态 | 单选 | 待审核 / 已采纳 / 已驳回 / 已入库 |
| 审核人 | 人员 | 负责审核的 PM |
| 审核意见 | 文本 | 采纳/驳回理由 |
| 入库位置 | 文本 | 采纳后实际写入的知识库章节/记录ID |
| 提交时间 | 创建时间 | 自动记录 |
| 提交人 | 文本 | 跑 `/需求共创` 的 PM |

#### 5.15.4 审核工作流

```
AI 生成沉淀候选
    │
    ▼
PM 确认提交（可逐条筛选）
    │
    ▼
写入 Bitable 暂存表（状态=待审核）
    │
    ▼
知识库负责人定期审核 ←─── 频率：每周一次或每 5 条触发
    │
    ├─ 已采纳 → 负责人（或 AI 辅助）写入对应知识库
    │              ├─ ①② → 更新飞书文档（新增/扩展章节）
    │              ├─ ③ → 新增 Bitable 记录（带三层关联）
    │              ├─ ④ → 新增/更新飞书文档
    │              └─ Skill → 提 prompt 更新需求
    │
    ├─ 已驳回 → 标注理由（个例/不准确/已有覆盖）
    │
    └─ 需讨论 → 拉相关 PM 讨论后再决定
```

**审核角色分工**：

| 归属知识库 | 审核人 | 理由 |
|-----------|--------|------|
| ① 需求框架库 | 高级PM / PM Lead | 框架层面的变更影响全团队 |
| ② 细节规范库 | 资深PM（有设计规范经验） | 需要判断设计模式的通用性 |
| ③ 业务规则库 | 合规对接PM + 合规团队 | 规则准确性需合规确认 |
| ④ 系统实现库 | 技术PM / 研发接口人 | 系统知识需技术确认 |
| Skill 更新 | Skill 维护者（你） | Prompt 变更需要测试验证 |

#### 5.15.5 入库路径

| 归属 | 入库方式 | 自动化程度 |
|------|---------|-----------|
| ① 需求框架库 | 负责人编辑飞书文档，新增/修改对应章节 | 手动（AI 可生成草稿） |
| ② 细节规范库 | 负责人编辑飞书文档，新增/修改对应规范条目 | 手动（AI 可生成草稿） |
| ③ 业务规则库 | AI 通过 Bitable MCP 写入新记录（负责人审核后触发） | 半自动 |
| ④ 系统实现库 | 负责人维护（当前未建） | 手动 |
| Skill prompt | 维护者更新 Skill 文件 + 测试 | 手动 |

**AI 辅助入库**：审核通过后，PM 可以用 `/需求沉淀` Skill 将已采纳条目自动写入知识库：
- 对 ③：直接调用 Bitable API 创建记录
- 对 ①②：生成符合文档格式的内容片段，PM 粘贴或 AI 通过 MCP 追加

#### 5.15.6 与 `/需求沉淀` Skill 的关系

四个 Skill 的完整闭环：

```
/需求研究 → /需求共创 → /PRD内审 → /需求沉淀
                │                        ▲
                └── Phase 5 沉淀候选 ────┘
```

- `/需求共创` Phase 5 负责**发现和提交**沉淀候选
- `/需求沉淀` 负责**审核和入库**——独立 Skill，支持：
  - 查看暂存表中待审核条目
  - 逐条审核（采纳/驳回/讨论）
  - 采纳后自动写入对应知识库
  - 统计沉淀数据（本月新增多少条、各知识库分布）

这样 `/需求共创` 每次运行都是知识库的"输入口"，`/需求沉淀` 是"质检关"。

#### 5.15.7 冷启动与渐进式沉淀

| 阶段 | 沉淀方式 | 说明 |
|------|---------|------|
| MVP | `/需求共创` Phase 4 末尾列出候选，PM 手动复制到 Bitable | 最小实现，验证沉淀价值 |
| V1 | AI 自动写入 Bitable 暂存表，飞书通知审核人 | 降低 PM 操作成本 |
| V2 | `/需求沉淀` Skill 上线，审核+入库全流程闭环 | 完整自动化 |
